Session 1

CLAUDE.md in Action

A single file that shapes every line the agent writes. Add one rule — change all future output.

  • Repository-level: team standards, architecture
  • Project-level: specific patterns for this app
  • User-level: personal preferences

Exercise: Write a CLAUDE.md for one of your current projects. 10 minutes.

# CLAUDE.md ## Tech Stack - Use PySpark for all data processing - All functions must have type hints - Tables use Unity Catalog: workshop_vibe_coding.<schema> ## Data Standards - Date columns: DATE, YYYY-MM-DD - Currency: DECIMAL(12,2) - All tables: processing_timestamp - Gold tables: partitioned by date ## Testing - pytest with PySpark fixtures - Small DataFrames (5-10 rows) - Assert schema, counts, values
Session 1

Anatomy of a Great PRD

Every effective PRD for agentic development has four essential sections.

1. Acceptance Criteria

Definition of Done

  • Specific, testable outcomes
  • "The API returns 200 with valid JSON"
  • "All tests pass with >90% coverage"
  • Agent knows exactly when to stop
2. Constraints

Non-Functional Requirements

  • Language, framework, version
  • "Use FastAPI, not Flask"
  • "Python 3.10+, PySpark 3.5"
  • Security, performance boundaries
3. Data Contracts

Interfaces & Schemas

  • Input/output schemas
  • Table schemas with column types
  • API request/response shapes
  • Error response formats
4. Examples

Concrete Input/Output

  • Sample inputs & expected outputs
  • "Given X, return Y"
  • Edge cases to handle
  • The more examples, the better

Garbage in, garbage out — at 100x speed. Precision matters more than ever.

  • Specific tech stack
  • Defined data contracts

Session 1

CLAUDE.md: Three Scope Levels

Instructions cascade from global to project-specific. More specific rules override general ones.

User-Level
~/.claude/CLAUDE.md

Personal Preferences

Your coding style, preferred tools, git conventions. "Always use uv run for Python." "Use imperative mood in commits."

▼
Repo-Level
./CLAUDE.md

Team Standards

Architecture decisions, testing patterns, dependencies. "Use PySpark for all data processing." "All tables in Unity Catalog."

▼
Project-Level
./src/CLAUDE.md

Module-Specific Rules

Component-specific patterns and constraints. "All API endpoints return JSON." "Use Pydantic models for validation."

Session 1

What to Put in Your CLAUDE.md

Include
  • Architecture decisions — "Medallion architecture: bronze/silver/gold"
  • Coding standards — "All functions must have type hints"
  • Testing patterns — "pytest with PySpark fixtures, 5-10 row DataFrames"
  • Data standards — "DATE type for dates, DECIMAL(12,2) for currency"
  • Dependencies — "Use databricks-sql-connector, not raw JDBC"
  • Deploy patterns — "Package with DABs, deploy via CI/CD"
Avoid
  • Obvious things — "Write clean code" (too vague)
  • Temporary context — "Fix the bug in line 42" (use prompts)
  • Contradictions — Conflicting rules confuse the agent
  • Overly rigid rules — Don't micromanage every decision
  • Stale info — Remove rules that no longer apply

Rule of thumb: If you've corrected the agent on the same thing twice, add it to CLAUDE.md.

Session 2

Writing Tests That Guide the Agent

Good tests are unambiguous specifications. They tell the agent exactly what "done" looks like.

def test_clean_transactions(): # GIVEN: 10 rows, 2 with null amounts raw = create_test_df([ ("store_1", "2024-01-01", 42.50), ("store_1", "2024-01-01", None), ("store_2", "2024-01-01", -5.00), # ... 7 more valid rows ]) # WHEN: we clean the data result = clean_transactions(raw) # THEN: invalid rows removed assert result.count() == 8 assert "transaction_amount" in result.columns assert result.filter( col("transaction_amount") <= 0 ).count() == 0

Key Principles

Given-When-Then

Structure every test with clear setup, action, and assertions.

Concrete Values

Use specific test data — "10 rows, 2 invalid" not "some rows."

Descriptive Names

Test names are documentation: test_removes_null_amounts

Multiple Assertions

Check schema, row counts, and specific values for completeness.

Session 2

Strategies to Improve Success Rates

1

Start Small: One Test, One Function

Don't dump 20 tests at once. Write one test, let the agent implement, verify, then add the next.

2

Provide Examples in Tests, Not Just Assertions

Test data is the best documentation. Create realistic sample DataFrames with comments explaining expected behaviour.

3

Use Descriptive Test Names as Documentation

test_removes_null_amounts_and_negative_values tells the agent exactly what the function should do.

4

Pin Critical Behaviour with Specific Value Assertions

Don't just check count() > 0. Assert exact values: revenue == 1234.56. Specific assertions catch subtle bugs.

5

Know When to Break the Loop

If the agent fails the same test 3+ times, intervene. Read the error, add context to your prompt, or simplify the test.

Session 2

Common TDD + Agent Pitfalls

Pitfall What Happens Fix
Tests too vague Agent writes code that technically passes but doesn't do what you want Add specific value assertions and edge cases
Too many tests at once Agent gets confused trying to satisfy everything simultaneously Write 1-3 tests, implement, then add more
Not running tests between changes Bugs accumulate, harder to debug which change broke things Run tests after each implementation step
Testing implementation, not behaviour Tests break on valid refactors, agent gets stuck Test inputs/outputs, not internal details
Missing edge cases Code works for happy path but fails on empty input, nulls, duplicates Add edge case tests after happy path passes
Ignoring test output Agent generates code, human doesn't verify test results Always read test output — it's your feedback loop
Session 2

The Test Pyramid for Data Engineering

Traditional software has 3 testing levels. Data pipelines need 5.

Full System + Client
Pipeline (incl. Service)
Multiple Jobs
Single Job
Unit / Component
↑ Fewer, slower, more realistic ↓ More, faster, more isolated

Key Insight

Traditional unit tests are fragile for data pipelines — the data changes rapidly, schemas evolve, and File I/O dominates the logic.

Recommendation

Test at the job-to-pipeline level where each job functions as a self-contained black box. Don't unit test File I/O or DataFrame append operations.

“I get paid for code that works, not for tests, so my philosophy is to test as little as possible to reach a given level of confidence.”

— Kent Beck

Session 2

Data as State: Why Data Testing Is Different

In software, tests run against code. In data engineering, you must also manage the data itself as test state.

The Three-Part Modification Burden

📄

Input Data

Test data must be representative of production data — schema, volume, edge cases

💻

Code

Transformation logic, joins, aggregations — the part we traditionally test

📊

Output Data

Expected results change whenever input data or code changes — triple the maintenance

Three Domain-Specific Complications

Traditional Databases

Monolithic pipelines, stateful tables, hard to replicate local test environments

Big Data Scale

Full production duplication is infeasible — can't just clone 10TB to run a test

Security & Obfuscation

Encryption and masking hamper verification — can't inspect PII columns in tests

Session 2

The Solution: Runtime Test Data Generation

Generate test data programmatically rather than copying from production.

# test_daily_aggregations.py
def create_test_transactions():
  """Generate known test data at runtime"""
  return spark.createDataFrame([
    ("STORE_001", "2024-01-15", 150.00),
    ("STORE_001", "2024-01-15", 75.50),
    ("STORE_002", "2024-01-15", 200.00),
  ], ["store_id", "date", "amount"])

def test_aggregation():
  df = create_test_transactions()
  result = compute_daily_agg(df)
  assert result.count() == 2 # 2 stores
  assert get_revenue("STORE_001") == 225.50

Why This Works

  • Easier refactoring — test data adapts with code changes
  • No privacy concerns — no PII leaves production
  • Scalable volumes — generate 10 rows or 10M rows
  • Edge case coverage — craft nulls, duplicates, future dates on purpose
  • Self-documenting — test data shows exactly what you're testing

Use Language-Native Frameworks

pytest + PySpark for Python. ScalaTest for Scala. JUnit for Java. The agent already knows these — lean into them.

This Is What Lab 1 Uses

You'll generate small DataFrames (5–10 rows) in pytest fixtures to test each pipeline function.

Where Human Input Is Essential

AI agents excel at implementation, but some decisions require human judgement.

Architecture & Design

Which patterns to use, how to structure modules, what to build vs buy. The agent implements your decisions.

Security Review

Authentication flows, access control, data exposure. Always review security-sensitive code manually.

Performance Strategy

Query optimisation, caching, partitioning. The agent can implement, but you choose the approach.

Business Logic

Domain rules, edge cases unique to your business, regulatory requirements. Only you know the context.

Edge Cases

The agent handles what you test for. Write tests for cases it hasn't seen — that's your unique value.

Code Review

Read what the agent generated. You're responsible for production code, regardless of who wrote it.

Session 3

Knowing When to Intervene

SignalWhat’s HappeningWhat to Do
Looping Same error 3+ times Agent lacks context or is stuck in a local minimum Stop. Provide the missing context or take a different approach entirely.
Hallucinating APIs Agent invents functions or methods that don’t exist Paste the actual API docs or point to the correct import path.
Over-engineering Simple task gets abstraction layers, design patterns, factories Redirect: “Keep it simple. No abstractions needed.”
Wrong architecture Correct code, wrong overall approach Stop early. Restate the architectural constraint in CLAUDE.md.
Slow convergence Tests pass but code quality is poor or non-idiomatic Add style rules to CLAUDE.md. Review and ask for specific refactors.
Security blind spot Agent doesn’t consider auth, injection, or data leaks Always review security-sensitive code yourself. Add security rules to CLAUDE.md.

The 3-strike rule: If the agent fails the same task 3 times, the problem is almost always in your prompt or missing context — not the model. Rewrite your instructions before trying again.
