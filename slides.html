THE META-FRAMEWORK

R.V.P.I. — One Loop for Directing Agents

Most AI-coding advice teaches R.P.I. — Research, Plan, Implement. V is missing. Retrieved context drifts; plans built on top of drift are confidently wrong.

PHASE 1

Research

Gather context — read codebase, recall past decisions, pull docs. No opinions yet.

PHASE 2 — RECURRING HABIT

Validate

Audit context. Ask at every transition:

  • • Is this current?
  • • Is this consistent?
  • • Is this trustworthy?
PHASE 3

Plan

Design the change. Success criteria, explicit steps. Built only on validated context.

PHASE 4

Implement

Execute in small steps. Validate after every step — not just at the end.

V is a habit, not a gate. It appears three times: after R (audit context — cheapest intervention point), after P (validate the plan before writing code), and after each I step (verify it worked). Skipping any one lets drift compound silently. Every technique in this block is a specialisation of one R.V.P.I. phase.

Why Specs Matter More with AI

Garbage in, garbage out — except now the garbage arrives in 30 seconds instead of 3 days.

BEHAVIOR-DRIVEN DEVELOPMENT

The BDD + Agent Workflow

Rule #1 applies here too: say what the system should do in plain English. The agent writes the test code. When a scenario fails, the agent reads the error and fixes itself.

01

Human Writes Features

Gherkin Given/When/Then — the spec in plain English. Use /bdd-features

02

Agent Implements Steps

Agent reads features, writes Python step definitions. Use /bdd-steps

03

Run & Iterate

Agent runs behave, sees failures, fixes. Use /bdd-run

04

Human Reviews

Add scenarios. Agent handles them. Ship it.

Critical: Always use two separate prompts. If the same prompt writes features AND steps, the agent writes steps that trivially pass rather than testing real behavior.

BEST PRACTICES

Why BDD Pairs Naturally With Agents

🎯

Agent Writes, You Read

The agent writes test code. But it’s Given/When/Then — you can read it and check if it captured what you meant.

🔄

Self-Correcting Loop

The agent runs behave, reads the traceback, fixes its step definitions, reruns. You watch.

🛡️

Acceptance Boundary

The agent can’t say “it works” when 3 of 8 scenarios are red. Structural proof, not verbal.

Pro tips: Start with one feature file. Use Background: for shared setup. Write declarative steps (“When I grant SELECT”) not imperative (“When I click the button”).

BEHAVIOR-DRIVEN DEVELOPMENT

Writing Gherkin That Guides the Agent

You describe what should happen. The agent writes the test. Because it’s Gherkin, you can read it back and verify it’s right.

@pipeline @smoke Feature: Transaction Data Cleaning As a data engineer I want invalid transactions removed So that downstream analytics are accurate Scenario: Remove null and negative amounts Given a raw transactions table with 10 rows And 2 rows have null amounts And 1 row has a negative amount When I run the clean_transactions pipeline Then the cleaned table should have 7 rows And no row should have a null amount And no row should have amount <= 0 And the schema should include "transaction_amount"

Key Principles

Declarative, Not Imperative

Describe what the system does, not how to click buttons.

Concrete Values

Use specific data — “10 rows, 2 invalid” not “some rows.”

One Behavior Per Scenario

If it tests two things, split it. Scenario names are documentation.

Use Backgrounds

Shared setup goes in Background: — don’t repeat connection steps.

Subagents, Skills, Hooks & Plugins

Once you’ve got the basics, these are your leverage multipliers. Four concepts that separate power users from first-time users.

Skills

  • Encoded domain knowledge, not prompts

Hooks

Circuit breakers, not just guardrails.

  • PreToolUse: block bad calls before they run
  • PostToolUse: enforce policy after calls
  • Stop: refuse to finish until clean

Plugins

Package for teams.

  • Bundle skills + agents + hooks

  • Shared team conventions, one install

VERIFICATION PATTERNS

Every Prompt Must Answer: How Will Claude Prove This Worked?

✅

Six Verification Patterns

Don’t ask “did this work?” — make the agent prove it with gates that either pass or fail.

  • BDD gates — write Gherkin features first, agent must pass all scenarios
  • Separate prompts — Prompt 1 writes tests, Prompt 2 implements
  • @dp.expect — data quality constraints that fail the pipeline
  • Schema contracts — StructType definitions as machine-readable specs
  • Full suite regression — run ALL tests after every change
  • Negative testing — test what should fail, not just what should pass
🧭

Explore First, Then Plan, Then Code

Separate research and planning from implementation to avoid solving the wrong problem fast.

  • Use /plan mode to align on approach first
  • Let the agent read the codebase before writing
  • Cheap to course-correct in planning, expensive in code

The difference: an agent that thinks it did the work vs one that proves it did. — Anthropic, Claude Code Best Practices

The Karpathy Experiment

Spent 4 hours refining an argument with an LLM. Genuinely convinced. Then asked the model to argue the opposite — it demolished his argument completely.

"The model was never reasoning toward truth. It was generating the most compelling version of whatever position it was asked to defend."

Your Defenses

Karpathy Test — Argue the Opposite

Before trusting any AI analysis, ask it to argue the opposite position. Especially useful when you don’t know the answer — if it demolishes its own argument, the original wasn’t reliable.

“Wait a minute…”

Two words that measurably improve critical evaluation. Prefixing with doubt triggers more rigorous analysis.

Structural Verification — Tests + Separate Prompts

Tests don’t rely on claims; code passes or it doesn’t. Prompt 1 writes tests, Prompt 2 implements. AI-authored code has 1.7× more issues, 2.7× more security vulnerabilities (Code Rabbit, 2026).

Treat Memory as Untrusted Input

Sycophancy has two modes: agent agrees with you now, or agrees with itself from a past session. The second is worse. Defence: set seeds, pin deps, version your CLAUDE.md; validate retrieved context before you plan on it.

This is not a bug to patch. RLHF training selected for agreement. The question: are you using AI to extend your thinking, or replace it? — François Chollet

SECURITY POSTURE · R.V.P.I. PHASE 2

Memory Is Untrusted Input

You already validate external API responses before you plan on them. Retrieved memory deserves the same posture.

The posture shift: Retrieved memory (CLAUDE.md, past session summaries, RAG results) is not authoritative by default — it’s accumulated text with no principled write path. A badly-formatted tool output, a hallucinated memory write, an auto-accepted suggestion — all persist across sessions and influence every future recall.

What changes if you adopt this posture

  • • Memory writes are explicit and logged, not automatic
  • • Retrieval results carry provenance metadata, not just relevance scores
  • • Conflicts between sources are surfaced, not silently resolved by ranking
  • • Human-in-the-loop sits at the governance layer, not just the output layer

Concrete defences for today

  • • After Lab 1 — re-read your CLAUDE.md. Is it still correct?
  • • Before planning a new change — verify the memory you’re building on
  • • Hooks as V-step automation: “is CLAUDE.md consistent with the current project?”
  • • When two memories disagree — surface the conflict; don’t let ranking decide

The third pillar: be Claude’s PM, own your harness, and be Claude’s librarian. Config files, retrieved memory, and skill metadata are library materials — catalogue them, retire stale ones, flag conflicts. That someone is the engineer.

PLATFORM RAILS · R.V.P.I. SURFACE

Where Databricks Absorbs V

V is expensive when the agent can break anything. On Databricks, whole categories of V are absorbed by the platform. Your V shrinks to the claims the rails can’t see.

Databricks absorbs (mechanical V)

  • • Unity Catalog — schema, permissions, lineage
  • • Lakeflow @dp.expect — data quality at runtime
  • • Serverless SDP — no broken cluster / partition config
  • • databricks bundle validate — deploy config before deploy
  • • Genie + semantic layer — grounds LLM queries to curated model
  • • Model Serving — versioning, rollback
  • • Lakebase — pool, OAuth token rotation
  • • Bundle targets — dev / prod isolation

What’s left for you (semantic V)

  • • CLAUDE.md claims — is what it asserts still true?
  • • Past-session memory — does it still hold?
  • • Cross-source consistency — do docs, code, past summaries agree?
  • • Intent-level correctness — SQL runs, but is the logic right?

The platform validates mechanism. You validate meaning.

Decision rule: Before you build a custom validator — a hook, a test, an MCP — check whether the platform already validates it deterministically. If UC, Lakeflow, or bundles catch it, you’re done. Spend your V budget on the claims no platform can see.

THE CENTRAL TECHNIQUE

Small Steps Beat Big Bang

The single biggest difference between ChatGPT users and Claude Code power users. Each prompt is one verifiable step — the rest follows.

Big Bang

What ChatGPT habits teach you.

# One mega-prompt "Build the entire bronze layer — both tables, all tests, quality expectations, and pipeline config."
  • 10–20 min wait with no visibility
  • Pile of code you didn’t watch generate
  • Multiple failures hit at once
  • You trust instead of verifying
  • Perfect setup for sycophancy

Small Steps

What Claude Code rewards.

# Five tight prompts 1. "Create the repo skeleton." 2. "Write @dp.table for CSV fetch." 3. "Write one schema test." 4. "Run the test. Fix if it fails." 5. "Add @dp.expect for validity."
  • 1–3 min per prompt, 10+ per hour
  • Active review between every step
  • Failures isolated to one change
  • You see the agent reason — you can disagree
  • Every prompt has its own verification

R.V.P.I. crossover

Small Steps is how Implement stays honest after V has done its job. Each prompt = one R.V.P.I. loop in miniature.

Heuristic: “After this prompt finishes, will I know whether it worked?” — if “no,” the prompt is too big. Split it.

Leaves vs trunks: small steps on leaf nodes (end features nothing depends on). Core architecture still gets human review. Anthropic calls this “leaf nodes only” — how they merged a 22K-line PR responsibly.

CONTEXT MANAGEMENT

What Are Tokens?

Tokens = the currency of AI

A token is roughly 3/4 of a word or ~4 characters. Not exactly words, not exactly characters — somewhere in between.

This sentence is about 8 tokens. ~8 tokens
A 200-line Python file ~2-3K tokens
A full README + 5 source files ~15-20K tokens
Harry Potter and the Philosopher’s Stone ~110K tokens

Two limits to know

200K Context window

How much the agent can “see” at once — your conversation, files read, tool results, CLAUDE.md. When it fills up, older content gets compacted (forgotten).

~1M/min Rate limit (shared)

Tokens per minute through AI Gateway — shared across all teams. If everyone reads large files at once, requests queue up.

Practical tip: Don’t ask the agent to “read everything in this directory.” Be specific. Smaller, focused requests = faster responses for everyone.

OPEN LAKEHOUSE

Managed Iceberg in Unity Catalog

Same UC governance. External engines read with zero copy via Iceberg REST Catalog.

The SQL

-- Same catalog, different format CREATE OR REPLACE TABLE workshop.gold.retail_summary_iceberg USING ICEBERG CLUSTER BY (state, month_date) AS SELECT * FROM workshop.gold.retail_summary; -- Readable by Snowflake, Trino, DuckDB, BigQuery -- via the Iceberg REST Catalog — no copy

Rules: no LOCATION, no PARTITION BY. Liquid clustering only. UC owns the path.

Pick Iceberg when

  • External engines will read the table
  • Vendor-neutral format required
  • Multi-cloud or multi-warehouse setup

HOW WE WORK · BLOCK 5

Pair Programming Norms

Two people, one keyboard. Fifteen-minute swaps. Navigator is not passive — Navigator is the V in R.V.P.I.

Driver

Navigator

Reads, verifies, challenges. Especially at the V step.

  • • Flag sycophantic agreement — argue the opposite

Escalation rule

Task doesn’t fit R.V.P.I. in 15 min? Split it.

APPENDIX

Subagents vs Agent Teams

The right question isn’t “should I use multiple agents?” — it’s “what kind of coordination does this task need?”

Subagents: Fire-and-Forget

Like delegating focused questions to researchers — they come back with distilled findings.

  • Own isolated context window
  • One job, then result returns to parent
  • Can’t talk to each other — by design
  • Key benefit: compression — vast exploration distilled to clean signal

Agent Teams: Ongoing Coordination

Like assembling a team that works in the same room — they persist, communicate, and coordinate.

  • Long-running instances with shared state
  • Peer-to-peer messaging between teammates
  • Shared task list with dependencies (blockedBy)
  • Key benefit: negotiation — discoveries in one thread change another

The #1 design principle: Start with a single agent. Push it until it breaks. That failure point tells you exactly what to add next.
