## Concepts

### New / sharpened this session

- **R.P.I. → R.V.P.I.** (Research → Validate → Plan → Implement). Alex Ker's original framework + the Validate step I proposed to fill the gap between Research and Plan. The V step is **context validation** — stale/conflicting/poisoned retrieved memory makes any plan worthless. (See `notes/rvpi-validate-step.md`.)
- **Memory governance as a security posture**: *"Retrieved memory should be treated as untrusted input, in the same way external API responses are treated as untrusted."* — architectural seam where trust, provenance, circuit breakers, and audit logging live. Cited mechanisms: trust-weighted retrieval, canonical state auditing, circuit breakers, immutable audit logging.
- **Memory poisoning** vs prompt injection — poisoning persists across sessions, propagates through RAG, doesn't need a sophisticated attacker (bad tool output / hallucinated memory write is enough).
- **Leaf nodes only** (Eric, Anthropic) — vibe code the tree's leaves where nothing depends on them; review the trunks. The 22K-line Anthropic RL PR landed responsibly with this discipline.
- **Harness as the unit of iteration** (Alex Ker) — the long-term asset isn't prompts; it's the CLAUDE.md + skills + hooks you iterate over sessions. *"Engineers used to argue about IDEs; now we argue about harnesses."*
- **Instruction budget** (Kyle @ HumanLayer) — LLMs follow only a few hundred instructions before entering a "dumb zone." ETH research: LLM-generated system prompts degrade performance AND cost ~20% more vs human-written.
- **Progressive disclosure** — descriptive file names so context loads on demand, not front-loaded. Claude Code ships with MCP tool indexing (85% context reduction); OpenCode/Codex load all MCP tools at session start.
- **Hooks as circuit breakers** (not just guardrails) — `PreToolUse` blocks before, `PostToolUse` enforces after, `Stop` refuses dirty finish. V-step automation when paired with R.V.P.I.

## Sharp edges

- **Facilitator file drift.** `slides.html` and `slides-facilitator.html` diverge subtly — different indentation, different separator comments, `data-notes` attribute. Every edit must be mirrored; I watched a previous session's `move_slides.py` silently fail to move slides in facilitator and leave orphans. **Solo pitfall: edit only one file, deck goes out of sync and you find out on workshop day.**
- **Slide renumbering is three places, not one.** Comment `<!-- SLIDE N: -->`, `aria-label="Slide N: ..."`, `<div class="slide-footer">N</div>`. Miss any of them and you get mismatched numbering that looks right until you land on the broken slide.
- **`data-notes` attributes don't auto-update when you edit slide content.** I sharpened the Hooks card on Power Tools but the speaker notes are still from the pre-edit version. A silent drift — would only surface when reading notes aloud during practice run.
- **Heredoc shell escaping mangles `<!--`.** Seen it twice — `bash -c "python3 -c '...' "` with `<!-- SLIDE` inside becomes `<\!-- SLIDE` and your regex silently doesn't match. Use `Write` tool for any Python containing HTML-comment literals.
- **Context poisoning is the failure mode nobody sees.** Agent adds a wrong rule to CLAUDE.md → follows it forever → every future session inherits it. No amount of R or P or I fixes this without an explicit V.
- **"Instruction budget" is counter-intuitive.** Adding more rules to CLAUDE.md to fix bad behaviour often makes behaviour worse — each extra rule crowds the attention budget and increases hallucination. Front-loading rules scales badly.
